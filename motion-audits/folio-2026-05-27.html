Primary: Jakub Krehel — Production Polish  |  Secondary: Emil Kowalski — Restraint & Speed  |  Selective: Jhey Tompkins — Delight

Folio has received several rounds of intentional polish — the drawer easing curve, the LikeButton micro-press, the BottomNav pill indicator, and the feed skeleton all signal real care. The foundations are solid. However, three systemic gaps undercut everything else: zero prefers-reduced-motion support (the entire animation surface is inaccessible to ~30% of users who need it), no exit animations anywhere (every modal disappears instantly, breaking spatial continuity), and ~12 JS-driven hover mutations that bypass CSS transitions entirely. Fix those three and the app immediately crosses a threshold from "polished demo" to "polished product." The existing pulsing indicators (flame, online dot, sparkles) are the clearest AI-slop tells in the codebase — they loop infinitely, serve no functional purpose, and would irritate daily users after the second week.

Issues to Address

🔴 Critical

Zero exit animations — every modal and sheet vanishes instantly

Every conditional render in Folio — BookDetailModal, NotificationsSheet, PostComments, FeedSkeleton→FeedView, previewPost, shareSession, and 57 more — unmounts with no exit transition. The sheet slides in gracefully (iOS curve, 320ms) then disappears in a single frame. This breaks the spatial model Jakub describes: exits orient users about where content "went."

All 63 conditional renders in App.jsx — key instances: App.jsx:10776 (BookDetailModal), App.jsx:10765 (NotificationsSheet), App.jsx:9438 (PostComments)

Fix: Add @keyframes slideDown and a dismissing CSS class triggered on close (before unmount), or introduce Framer Motion AnimatePresence + exit prop on primary sheets.

The sheet demo shows: enter → hold → exit should match the enter curve but reversed at subtler scale.

🔴 Critical

12 JS hover mutations bypass CSS transitions — instant visual snaps

onMouseEnter/onMouseLeave handlers directly mutate element.style.backgroundColor, element.style.transform, and element.style.boxShadow with no transition on the element. These snap instantaneously. Jakub's rule: even 150ms transitions feel dramatically more polished than instant changes. The pattern is also unmaintainable — state lives in DOM, not CSS.

App.jsx:2383, 3659–3662, 4269–4270, 6335–6336, 7304–7305, 7351–7352, 7476–7477, 8334–8335, 8424–8425, 8488–8489

Fix: Move each to a CSS class with transition: background-color 150ms ease. Apply via className toggle or CSS :hover behind @media (hover: hover) and (pointer: fine).

🟡 Important

Icon swaps lack blur — state transitions read as snaps

The LikeButton correctly uses dual-opacity for the heart swap, but skips filter: blur() during the crossfade. Per Jakub: "Blur bridges the visual gap by blending two states together, tricking the eye into perceiving a single smooth transformation." The same issue applies to the share button's "Compartir → ¡Publicado! ✓" state swap (line 8682) which changes text content instantly with only transition: background 0.2s.

App.jsx:7907–7914 (LikeButton hearts), App.jsx:8682 (share button text)

Fix: Add filter: blur(3px) at the midpoint of the opacity transition. For LikeButton: style={{ transition: "opacity 120ms ease, filter 120ms ease" }} with filter: blur(3px) at 0.5 opacity.

Issues to Address

🔴 Critical

Zero prefers-reduced-motion support — entire animation surface is inaccessible

Not a single @media (prefers-reduced-motion: reduce) rule exists anywhere in the codebase — neither in src/index.css nor in the inline <style> blocks. Four animations loop infinitely without any pause mechanism: fire-pulse-anim (streak flame, 1.4s), onlinePulse (friend dot, 2.5s), shimmer (skeleton, 1.5s), sparkle-pulse (Sparkles icon, 2.2s). This is a WCAG 2.1 Level AAA violation (Success Criterion 2.3.3). Approximately 30% of users have some motion sensitivity; the flame icon loops on the most prominent element of the primary screen.

src/index.css (all keyframes), App.jsx:10705–10749 (inline keyframes incl. fire-pulse-anim, sparkle-pulse)

Fix: Add a single block at the end of src/index.css:
@media (prefers-reduced-motion: reduce) { *, *::before, *::after { animation-duration: 0.01ms !important; animation-iteration-count: 1 !important; transition-duration: 0.01ms !important; } }

🟡 Important

transition: "all" remains in 15+ locations

Previous rounds of Emil's fixes cleaned the most visible instances, but 15+ remain. transition: all forces the browser to monitor every animatable CSS property — including expensive layout-triggering ones like padding, margin, height, width, and border-width — on every frame. On a mobile device scrolling through a feed, this compounds. The desktop nav buttons (line 1600), status pill buttons (line 2305), friend search items (line 5486, 5523), and notification list items (line 7163) are the worst offenders since they're triggered repeatedly.

App.jsx:1600, 2305, 3615, 3878, 5486, 5523, 7163 — plus Tailwind transition-all on 9 buttons

Fix: Replace each with specific properties. Nav buttons: transition: "background-color 150ms ease, color 150ms ease". Status pills: transition: "background-color 120ms ease, color 120ms ease, border-color 120ms ease".

🟡 Important

Progress bars animate height / width — triggers layout on every frame

Four progress bars across the app (reading score, friend comparison bars, session history bars) use transition: "width 0.4s ease" or transition: "height 0.4s ease". Animating height and width forces browser layout recalculation on every animation frame — the most expensive path in the rendering pipeline. On a mid-range Android device scrolling through profile stats, this can drop frames. The fix is transform: scaleY() + transform-origin: bottom, which runs entirely on the GPU compositor thread.

App.jsx:5706, 5710 (reading comparison bars), App.jsx:5838–5839 (session history chart), App.jsx:6099 (reading score bar), App.jsx:4242 (achievement progress)

Fix: Replace width: \`${pct}%\` with transform: \`scaleX(${pct / 100})\` and transformOrigin: "left center". For vertical bars: scaleY + transformOrigin: "bottom".

🟡 Important

Inconsistent hover-scale across 5+ elements — borderline scatter slop

Five different scale values are used for hover across the app: hover:scale-105 (add button), hover:scale-[1.02] (friend accept), hover:scale-[1.01] (form submit buttons), hover:-translate-y-0.5 (mode cards), hover:scale-110 (star rating). None are gated with @media (hover: hover) and (pointer: fine), so touch users get false hover activations on tap. There's no consistent system — each component made an independent choice. Per Jhey: building diverse demos trains you to switch contexts, but a production app needs a system, not experiments.

App.jsx:1431 (star, scale-110), 1504 (scale-[1.02]), 1814 (scale-105), 2704, 2798 (scale-[1.01]), 2539, 2544 (translate-y-0.5)

Fix: Standardize to one hover scale value (scale(1.02)) for interactive cards, guard with @media (hover: hover) and (pointer: fine), and remove hover:scale-110 from the star rating entirely (use opacity instead).

Opportunities

  • 💡 Celebration modal — clip-path book reveal — The bookAppear animation scales and rotates the cover. A clip-path: inset() reveal from top-to-bottom would feel more like unwrapping — more cinematic for a rare, celebratory moment. Uses GPU-compositor-only path. — App.jsx:10129
  • 💡 Progress bars — @property + linear() spring fill — Register a CSS @property for the progress value and animate it with linear() for a physics-based fill that doesn't require JS. Eliminates the layout-cost issue in Finding 6 and adds personality. — App.jsx:4242, 6099
  • 💡 BottomNav active pill — @property position animation — The current pill uses background-color transition. An absolutely-positioned pill that physically moves between tabs (using @property or a shared layout element) would feel more physical, like a real selection indicator. — App.jsx:1644–1674
  • 💡 Streak milestone — burst particle moment — When a user logs their first day of a new streak (or hits day 7/30), a brief clip-path burst or confetti pop would feel appropriate. Rare event, high emotional resonance. — App.jsx:8181

Through Jhey's lens: The celebration animations (bookAppear, achieveIconIn, heartPop, confetti) show a team that understands delight has a place in Folio — and they've placed it correctly, at rare moments. The three infinite pulsing indicators are the main creative misstep: they're attempting delight at the wrong frequency (infinite) and the wrong scale (tiny status dots). Remove those, invest that energy in the celebration moments, and Folio's motion personality becomes coherent.
